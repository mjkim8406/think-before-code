# Agent.md

## 목적

이 앱은 코딩테스트 문제를 **맞히는 앱**이 아니라, 문제를 보고 **어떤 개념을 떠올려야 하는지**, **출제 의도가 무엇인지**, **라이브코딩처럼 어떻게 설명해야 하는지**를 훈련하는 *이다.

핵심 목표는 다음 3가지다.

1. 문제를 보고 적절한 알고리즘/자료구조 후보를 식별한다.
2. 왜 그 접근을 선택해야 하는지 설명할 수 있다.
3. 코드를 짜기 전에 풀이 흐름을 구조화할 수 있다.

---

## 제품 한 줄 정의

하루 1문제씩 코딩테스트 문제를 읽고,

* 어떤 개념을 써야 하는지,
* 어떤 조건을 봐야 하는지,
* 면접처럼 어떻게 풀이를 설명해야 하는지
  훈련하는 사고력 중심 모바일 앱.

---

## 타겟 사용자

* 코딩테스트를 준비하지만 꾸준히 못하는 사람
* 문제를 풀어도 유형 분류가 잘 안 되는 사람
* 코드 작성 전에 설계가 약한 사람
* 라이브코딩/기술면접에서 풀이 설명이 약한 사람

---

## 풀고자 하는 문제

기존 코테 학습 방식의 문제는 아래와 같다.

* 문제를 보자마자 구현부터 하려 한다.
* 이 문제가 어떤 유형인지 분류하지 못한다.
* 왜 이 알고리즘을 써야 하는지 설명하지 못한다.
* 제출 결과만 보고 학습이 끝난다.
* 문제를 꾸준히 푸는 루틴이 잘 형성되지 않는다.

이 앱은 이를 다음 방식으로 해결한다.

* 하루 1문제로 선택 피로를 줄인다.
* 문제 해석 체크리스트로 사고를 강제한다.
* 해설은 정답이 아니라 사고 흐름 중심으로 제공한다.
* 정답률이 아니라 해석력과 설명력을 기록한다.

---

## 핵심 원칙

### 1. 제출보다 해석이 먼저다

초기 버전에서는 실제 코드 제출/채점보다 아래를 더 중요하게 본다.

* 문제 유형 식별
* 핵심 제약 파악
* 접근 후보 비교
* 시간복잡도 판단
* 설명 흐름 구성

### 2. 선택지를 줄인다

사용자는 스스로 문제를 고르지 않는다.
앱이 오늘의 문제를 1개 할당한다.

### 3. 사용자는 ‘생각’을 반드시 남겨야 한다

단순히 해설만 읽는 구조는 금지한다.
해설을 보기 전에 최소한 아래는 작성해야 한다.

* 예상 유형
* 왜 그렇게 생각했는지
* 버려야 할 접근
* 최종 풀이 방향

### 4. 결과보다 사고 과정을 저장한다

`맞았다/틀렸다`보다 아래 데이터가 더 중요하다.

* 개념을 맞췄는가
* 출제 의도를 맞췄는가
* 풀이 순서를 설명할 수 있는가
* 엣지 케이스를 고려했는가

---

## 사용자 흐름

### 1. 홈

사용자는 홈에서 오늘의 문제 1개를 본다.

표시 정보:

* 오늘의 문제
* 현재 streak
* 최근 약한 개념
* 오늘의 목표

### 2. 문제 읽기

문제를 읽는다.
이 단계에서는 해설을 절대 바로 보여주지 않는다.

### 3. 문제 해석

앱이 아래 질문을 순서대로 던진다.

* 이 문제는 어떤 유형인가?
* 가장 먼저 봐야 하는 조건은 무엇인가?
* 시간복잡도상 불가능한 접근은 무엇인가?
* 어떤 자료구조/알고리즘 후보가 있는가?
* 출제자는 무엇을 보려는가?

### 4. 풀이 흐름 정리

사용자는 라이브코딩 설명처럼 풀이 흐름을 작성한다.

* 문제를 한 문장으로 요약
* 브루트포스 가능 여부
* 최종 접근 방식
* 필요한 자료구조
* 시간복잡도
* 주의할 엣지 케이스

### 5. 해설 비교

사용자의 생각과 모범 사고 흐름을 비교한다.

해설 구조:

* 먼저 봐야 하는 포인트
* 개념 후보
* 최종 선택 이유
* 흔한 오답 접근
* 실수 포인트
* 면접식 설명 예시

### 6. 회고 저장

최종적으로 아래를 저장한다.

* 개념 적중 여부
* 의도 적중 여부
* 설명 완성 여부
* 사용자가 느낀 난이도

## 핵심 화면

### 홈 화면

목적: 오늘 학습 시작을 가장 쉽게 만든다.

포함 요소:

* 오늘의 문제 카드
* 스트릭 카드
* 최근 약점 개념 카드
* 바로 시작 버튼

### 문제 해석 화면

목적: 문제를 보고 곧바로 구현하지 못하게 막고, 먼저 사고하게 만든다.

포함 요소:

* 문제 본문
* 제약 조건
* 유형 선택
* 핵심 포인트 작성
* 버릴 접근 선택

### 풀이 흐름 화면

목적: 라이브코딩 설명처럼 사고를 말의 흐름으로 정리하게 한다.

포함 요소:

* 문제 요약
* 접근 후보
* 최종 접근
* 시간복잡도
* 엣지 케이스

### 해설 비교 화면

목적: 정답 확인이 아니라 사고 방식의 차이를 비교하게 한다.

포함 요소:

* 내 답변
* 모범 사고 흐름
* 틀리기 쉬운 접근
* 면접식 설명 예시

### 복습 화면

목적: 틀린 문제가 아니라 약한 사고 패턴을 복습하게 한다.

포함 요소:

* 자주 틀리는 개념
* 자주 놓치는 포인트
* 재도전 문제 목록

## 데이터 모델 초안

### Problem

문제 자체와 해석 메타데이터를 가진다.

주요 필드:

* id
* title
* description
* difficulty
* conceptTags
* secondaryConceptTags
* intentDescription
* keyObservation
* wrongApproach
* liveCodingFlow

### ProblemTrainingSession

사용자의 하루 학습 기록이다.

주요 필드:

* id
* userId
* problemId
* selectedConcepts
* whyThisConcept
* problemSummary
* bruteForceAnalysis
* finalApproach
* timeComplexity
* edgeCases
* conceptMatched
* intentMatched
* flowCompleted
* trainingDate

### UserWeakConcept

사용자의 반복 약점을 기록한다.

주요 필드:

* id
* userId
* conceptTag
* missCount
* lastMissedAt

### UserStreak

연속 학습 상태를 기록한다.

주요 필드:

* userId
* currentStreak
* lastTrainedDate

## 문제 메타데이터 설계 원칙

이 앱의 핵심은 문제 원문보다 문제 메타데이터다.

각 문제에는 반드시 아래 정보가 있어야 한다.

* 대표 개념
* 보조 개념
* 출제 의도
* 가장 먼저 봐야 하는 조건
* 흔한 오답 접근
* 면접식 풀이 설명 예시

해설을 자유서술로 두지 말고 반드시 템플릿화한다.

해설 템플릿:

1. 이 문제를 보면 먼저 봐야 하는 것
2. 가능한 접근 후보
3. 왜 이 접근을 선택하는가
4. 버려야 할 접근
5. 사용할 자료구조
6. 시간복잡도
7. 실수 포인트
8. 면접식 설명 예시

## 추천 사용자 액션 규칙

앱은 사용자를 많이 생각하게 하되, 많이 입력하게 만들면 안 된다.

입력 UX 원칙:

* 최대한 객관식 + 짧은 서술
* 긴 에세이 입력 금지
* 한 화면에 질문 1개 또는 2개만 노출
* 해설은 반드시 입력 이후에만 공개
